## 1. 架构总览与设计原则

### 1.1 产品定位

本平台定位为**面向国内电商环境的数字人直播带货 SaaS 平台**，为商家提供从"选品→形象定制→脚本生成→直播间搭建→多平台推流→实时互动→数据分析"的全链路服务。自身不建设直播平台，而是作为第三方技术服务商与各主流直播平台（淘宝、抖音、京东、快手、拼多多等）对接。

### 1.2 核心设计原则

| 原则 | 说明 | 架构体现 |
|------|------|---------|
| **国内优先 (China-First)** | 技术栈和平台对接优先服务国内电商环境 | DeepSeek/Qwen LLM、CosyVoice2 TTS、淘宝/抖音/京东 API 优先适配 |
| **云原生 (Cloud-Native)** | 全量容器化、声明式部署、不可变基础设施 | K8s + Docker + Helm + ArgoCD |
| **弹性伸缩 (Elastic)** | 基于流量的自动扩缩容，GPU 资源池化 | HPA/VPA + GPU 节点池 + 播单推流模式降本 |
| **高可用 (HA)** | 核心链路多副本、自动故障转移、优雅降级 | 多 AZ 部署、主从切换、降级为预录播单 |
| **安全合规 (Compliance)** | 满足国内直播电商监管要求，三级风控体系 | AI 标识、内容审核、数据留存 ≥ 30 天 |
| **可观测 (Observable)** | 全链路追踪、指标监控、日志聚合 | Prometheus + Grafana + Jaeger + ELK |
| **模块可替换 (Pluggable)** | AI 引擎、平台适配器可独立升级/替换 | 适配器模式 + 服务网格 + 接口抽象 |
| **渐进式演进 (Evolutionary)** | POC → MVP → 工业级，分阶段交付 | Phase 1-3 分期规划，平滑迁移 |

> **竞品对标：** 以下设计原则的制定基于与 Johnsmith.ai（头部品牌零失误）、特看科技（形象生成强项）、元真数字人（低价基础款）的差异化分析。我们的架构目标不是在所有维度上超越每一个竞品，而是在 **四个核心差异化轴** 上建立壁垒：用户画像驱动对话、多平台主数据同步、自研情感 TTS、专注电商带货转化率。

### 1.2.1 竞品对标与差异化架构决策

| 差异化轴 | 竞品现状 (架构盲区) | 我们的架构决策 | 对应章节 |
|---------|-------------------|---------------|---------|
| **★ 用户画像驱动对话** | 竞品千人一面 FAQ 匹配，Johnsmith 有意不做个性化（怕品牌投诉） | 互动管线内置 User Profile 环节 (Redis 热数据 + PG 冷数据)，画像 JSON 注入 LLM Prompt，实现千人千面回复 | [第5章](#5-实时互动管线--核心运行时引擎) |
| **★ 多平台主数据同步** | **全行业盲区**：竞品只做推流，商品/库存/订单让商家自己管 | Canonical Product Model + Platform Adapter 模式，16 个微服务中独立 `platform-sync-svc`，一处管理一键同步全平台 | [第8章](#8-多平台适配器架构) |
| **★ 自研情感 TTS** | 多数方案语音机械，竞品多用第三方 TTS（无法定制情感） | CosyVoice2 + TensorRT 自研，10 模块管线 (M1-M10)，5 种情感 + 3 种风格 + <200ms 首包 | [第7章](#7-tts-语音合成引擎--深度设计) |
| **★ AI 文字场控** | Johnsmith 有评论区智能场控但仅服务头部 KA；特看/元真无此能力 | 纯文本 LLM 驱动，独立于语音播报的第二输出通道，成本极低（无需 TTS/渲染），腰部商家可负担 | [5.4 节](#54-ai-文字场控子系统) |
| **专注电商带货** | 竞品做泛场景（直播+短视频+客服+教育），架构臃肿 | 所有微服务以 GMV 转化率为核心指标，播单引擎 + 互动管线深度耦合电商场景 | [第4章](#4-微服务架构设计) |

### 1.3 技术栈选择：国内优先

| 技术域 | 首选方案 | 备选方案 | 选型理由 |
|--------|---------|---------|---------|
| **LLM 推理 (Phase 1)** | **DeepSeek API** (deepseek-chat) | 通义千问 API | 性价比最高、中文能力强、延迟可控 |
| **LLM 推理 (Phase 2)** | **Qwen-3-7B + LoRA 微调** 自部署 | DeepSeek-V3-Lite | 开源可商用、中文电商领域适配好 |
| **TTS 引擎** | **CosyVoice2** (阿里通义实验室) | FishSpeech / GPT-SoVITS | 零样本克隆最优、中文 MOS > 4.2、开源 |
| **口型同步** | **Wav2Lip** + 自研优化 | NVIDIA Audio2Face | 开源成熟、实时性好、社区活跃 |
| **向量数据库** | **Milvus** (国产开源) | Elasticsearch 向量插件 | 性能领先、社区活跃、国产替代 Pinecone |
| **后端框架** | **Python FastAPI** (统一技术栈) | Go (高并发热点优化) | AI 生态天然 Python；统一语言降低团队成本 |
| **对象存储** | **MinIO** (私有部署) / **阿里云 OSS** | AWS S3 | 私有部署降本，阿里云适配国内网络 |
| **消息队列** | **Kafka** (开源) | RocketMQ (阿里系) | 业界标准、生态完善、高吞吐 |
| **关系数据库** | **PostgreSQL 15** + TimescaleDB | MySQL 8.0 | 功能丰富、JSONB 支持、时序扩展 |
| **缓存** | **Redis 7 Cluster** | — | 业界标准 |
| **搜索引擎** | **Elasticsearch 8** | — | 业界标准 |
| **容器编排** | **Kubernetes** (K8s) | — | 业界标准 |
| **服务网格** | **Istio** + Envoy | — | 流量管理、可观测、安全 |
| **CI/CD** | **GitHub Actions** + **ArgoCD** | GitLab CI + Jenkins | GitOps 工作流、声明式部署 |
| **监控** | **Prometheus** + **Grafana** | — | 业界标准 |
| **日志** | **ELK** (Elasticsearch + Logstash + Kibana) | 阿里云 SLS | 自建可控 |
| **链路追踪** | **Jaeger** | SkyWalking (国产) | OpenTracing 标准 |
| **GPU 推理加速** | **TensorRT-LLM** + **vLLM** | ONNX Runtime | 推理吞吐最大化 |

> **决策说明：** 基础中间件（PostgreSQL、Redis、Kafka、K8s）采用国际开源标准组件，这些是全球通用的基础设施，不存在国内/国外之分。AI 引擎层（LLM、TTS、向量库）优先使用国内技术方案或国产开源项目——DeepSeek/Qwen 作为 LLM、CosyVoice2 作为 TTS、Milvus 作为向量库。对象存储在国内部署场景优先选用阿里云 OSS（网络延迟低），私有部署使用 MinIO。

### 3.2 层级职责与通信规则

| 层级 | 职责 | 可调用 | 被调用 |

| L1 接入层 | 流量接入、协议转换、安全防护 | L2 | 外部客户端 |
| L2 业务服务层 | 业务逻辑、CRUD、工作流编排 | L3, L5 | L1 |
| L3 AI 中台层 | AI 模型推理、GPU 调度 | L5, L6 | L2, L4 |
| L4 互动引擎层 | 实时弹幕→回复→播报管线 | L3, L5, L6 | L2 |
| L5 消息与编排层 | 异步解耦、事件驱动、服务发现 | L6 | L2, L3, L4 |
| L6 数据层 | 持久化、缓存、检索 | — | L2-L5 |
| L7 基础设施层 | 计算、网络、存储、监控 | — | 全层 |

> **通信规则：** 同层服务间通过 gRPC 同步调用 + Kafka 异步事件。跨层调用遵循"上层调用下层，下层通过事件回调上层"原则。L5（消息层）作为横向解耦层，所有异步事件通过 Kafka 流转。

### 5.2 端到端延迟预算

#### 5.2.1 语音播报声道

| RAG 检索 | Milvus 向量搜索 | 100ms | 700ms |

| TTS 首包 | CosyVoice2 | 200ms | 1700ms (P1) / 1400ms (P2) |

| **端到端总计** | | | **< 2s ✅** |

#### 5.2.2 文字评论声道

| 阶段 | 组件 | 预算 | 累计 |
|------|------|:----:|:----:|
| 弹幕采集 → NLP | 同语音声道 | 600ms | 600ms |
| 声道路由决策 | 规则引擎 (本地) | < 5ms | 605ms |

> **关键洞察：** 文字声道省去了 TTS (200ms)、口型渲染 (100ms)、RTMP 推流 (100ms) 三个环节，且文字回复无需排队等语音播报完成，可**并行输出**。简单问题文字回复的实际用户感知延迟可低至 1s 内。

### 5.3 互动优先级与声道路由

```
弹幕优先级 (按商业价值排序) + 声道路由决策:

P0 — 下单意向: "怎么买""多少钱""链接在哪"
     → 语音: 立即打断当前片段,插播回复,延迟 < 1.5s
     → 文字: 同步弹商品卡片 + 优惠券 (弹袋/弹券)

P1 — 商品提问: "什么面料""有没有大码"
     → 语音: 等待当前片段结束,优先回复
     → 文字: 不额外发送 (语音回复已覆盖)

P2a — 简单咨询: "有优惠吗""包邮吗""今天发什么"
     → 语音: 不占用语音播报 (节省 TTS + 渲染成本)
     → 文字: 评论区直接文字回复,延迟 < 1s

P2b — 互动/寒暄: "主播好美""早上好""支持主播"
     → 语音: 合并批量处理或跳过
     → 文字: 场控账号批量回复 (感谢/表情),营造活跃氛围

P3 — 氛围触发 (主动): 在线人数低 / 互动冷清 / 商品切换
     → 语音: 不触发
     → 文字: 氛围小助理自动发送互动话题 ("大家喜欢什么颜色?")

P4 — 无关/骚扰/负面: 纯表情/刷屏/辱骂
     → 语音: 忽略
     → 文字: 忽略 / 隐藏 / 安抚回复 (FC-05)

声道路由核心规则:
┌──────────────────────────────────────────────────────────┐
│  ★ 语音通道 (稀缺资源,需排队,成本高):                       │
│     仅用于 P0 下单意向 + P1 商品提问                        │
│                                                          │
│  ★ 文字通道 (低成本,可并行,无需 TTS/渲染):                  │
│     覆盖 P2a 简单咨询 + P2b 互动寒暄 + P3 氛围触发           │
│                                                          │
│  ★ 双通道并行 (P0):                                       │
│     语音回复 + 同时弹商品卡/优惠券                          │
└──────────────────────────────────────────────────────────┘

降级策略:
┌──────────────────────────────────────────────────────────┐
│  LLM 超时 (> 2s)      → 降级为 FAQ 精确匹配               │
│  LLM 故障/不可用       → 降级为预设安全回复模板             │
│  TTS 故障             → 播报预录音频 (兜底片段)            │
│  Redis/Milvus 不可用  → 跳过画像+知识增强,仅用商品上下文    │
│  文字场控 LLM 故障     → 降级为预置话术模板轮播              │
│  整体互动故障          → 纯播单模式继续推流 (直播不中断)     │
│  平台评论 API 不可用   → 文字场控暂存队列,恢复后补发         │
└──────────────────────────────────────────────────────────┘
```

```
弹幕优先级 (按商业价值排序) + 声道路由决策:

P0 — 下单意向: "怎么买""多少钱""链接在哪"
     → 语音: 立即打断当前片段,插播回复,延迟 < 1.5s
     → 文字: 同步弹商品卡片 + 优惠券 (弹袋/弹券)

P1 — 商品提问: "什么面料""有没有大码"
     → 语音: 等待当前片段结束,优先回复
     → 文字: 不额外发送 (语音回复已覆盖)

P2a — 简单咨询: "有优惠吗""包邮吗""今天发什么"
     → 语音: 不占用语音播报 (节省 TTS + 渲染成本)
     → 文字: 评论区直接文字回复,延迟 < 1s

P2b — 互动/寒暄: "主播好美""早上好""支持主播"
     → 语音: 合并批量处理或跳过
     → 文字: 场控账号批量回复 (感谢/表情),营造活跃氛围

P3 — 氛围触发 (主动): 在线人数低 / 互动冷清 / 商品切换
     → 语音: 不触发
     → 文字: 氛围小助理自动发送互动话题 ("大家喜欢什么颜色?")

P4 — 无关/骚扰/负面: 纯表情/刷屏/辱骂
     → 语音: 忽略
     → 文字: 忽略 / 隐藏 / 安抚回复 (FC-05)

声道路由核心规则:
┌──────────────────────────────────────────────────────────┐
│  ★ 语音通道 (稀缺资源,需排队,成本高):                       │
│     仅用于 P0 下单意向 + P1 商品提问                        │
│                                                          │
│  ★ 文字通道 (低成本,可并行,无需 TTS/渲染):                  │
│     覆盖 P2a 简单咨询 + P2b 互动寒暄 + P3 氛围触发           │
│                                                          │
│  ★ 双通道并行 (P0):                                       │
│     语音回复 + 同时弹商品卡/优惠券                          │
└──────────────────────────────────────────────────────────┘

降级策略:
┌──────────────────────────────────────────────────────────┐
│  LLM 超时 (> 2s)      → 降级为 FAQ 精确匹配               │
│  LLM 故障/不可用       → 降级为预设安全回复模板             │
│  TTS 故障             → 播报预录音频 (兜底片段)            │
│  Redis/Milvus 不可用  → 跳过画像+知识增强,仅用商品上下文    │
│  文字场控 LLM 故障     → 降级为预置话术模板轮播              │
│  整体互动故障          → 纯播单模式继续推流 (直播不中断)     │
│  平台评论 API 不可用   → 文字场控暂存队列,恢复后补发         │
```

### 6.2 LLM 分阶段策略

#### Phase 1: DeepSeek API (MVP)

```

```

#### Phase 2: Qwen-3-7B LoRA 微调 + 自部署

```
  简单问题 → 本地 Qwen-3-7B-LoRA
```

### 7.3 TTS 引擎模块接口定义

```protobuf
service TTSService {
  // 流式合成 (核心接口)
  rpc SynthesizeStream(SynthesizeRequest) returns (stream AudioChunk);

  // 批量预合成 (开播前预热)
  rpc PreSynthesize(PreSynthesizeRequest) returns (PreSynthesizeResponse);

  // 音色管理
  rpc CreateVoice(CreateVoiceRequest) returns (Voice);
  rpc GetVoice(GetVoiceRequest) returns (Voice);
  rpc ListVoices(ListVoicesRequest) returns (ListVoicesResponse);
  rpc DeleteVoice(DeleteVoiceRequest) returns (Empty);

  // 音色克隆
  rpc StartVoiceClone(StartVoiceCloneRequest) returns (VoiceCloneTask);
  rpc GetVoiceCloneStatus(GetVoiceCloneStatusRequest) returns (VoiceCloneTask);

  // 健康检查
  rpc Health(HealthRequest) returns (HealthResponse);
}

message SynthesizeRequest {
  string text = 1;              // 待合成文本 (≤ 2000 字符)
  string voice_id = 2;          // 音色 ID
  string emotion = 3;           // excited/warm/enthusiastic/calm/urgent
  float speed = 4;              // 语速 0.5-2.0
  string request_id = 5;        // 请求追踪 ID
  bool skip_cache = 6;          // 跳过缓存，强制推理 (实时互动场景)
  LiveContext live_context = 7; // 直播上下文 (用于 Prosody 优化)
}

message AudioChunk {
  string request_id = 1;
  int32 sequence = 2;           // 从 0 递增
  bytes pcm_data = 3;           // 320 bytes = 20ms @ 16kHz mono s16le
  int32 sample_rate = 4;        // 16000
  float rtf = 5;                // 当前实时率
}
```

### 7.4 TTS 缓存策略

```
缓存层级:

L1 — GPU 显存缓存 (最快, ~1ms)
    • 当前正在直播的音色 Embedding
    • 容量: 约 20-50 个音色 (受显存限制)

L2 — Redis 音频缓存 (快, ~5ms)
    • 高频话术完整 PCM: "欢迎""感谢""多少钱""包邮吗"等
    • TTL: 24h
    • 预缓存: 开播前预热 Top-50 话术

L3 — MinIO/OSS 持久化 (慢, ~100ms)
    • 历史合成音频归档
    • 用于离线分析和模型优化

缓存键规则:
  cache_key = md5(normalized_text):voice_id:emotion:speed

预缓存策略:
  开播前 5 分钟:
    1. 获取直播间 Top-50 高频话术列表
    2. 并行调用 TTS 合成 (优先级低于在线请求)
    3. 写入 Redis L2 缓存
    4. 开播后优先命中缓存
```

## 15. 全人克隆管线架构 (Phase 3 核心差异化能力)

### 15.1 设计目标

全人克隆是本平台的终极差异化能力——商家上传一段视频，系统在 24 小时内生成形象+音色+表情+动作一体化的数字分身。在合规 AI 标识的前提下，观众主观感受"跟真人主播几乎没区别"。

| 维度 | Phase 2 基线 | Phase 3 目标 |
|------|-------------|-------------|
| **形象** | 2D GAN + FaceFormer，面容还原度 > 99% | 3DGS/NeRF 实时渲染，面容还原度 > 99%，支持多角度 |
| **音色** | CosyVoice2 full-training（15min，24h） | Few-shot（3-10s，分钟级），相似度 > 0.90 |
| **表情** | 情感→BlendShape 映射（5 种情感） | 微表情克隆（298 种），从视频自动提取 |
| **动作** | 预设动作库 + 关键词触发 | 视频动作提取 + 风格迁移 + 语音驱动生成 |
| **渲染** | 播单模式（预渲染 MP4 分段推流） | 实时合成模式（边生成边推流），自然度质的飞跃 |
| **GPU** | 2×L40S (render+TTS) | 4×L40S (3DGS render + TTS + LLM + clone training) |

### 15.3 3D Gaussian Splatting 实时渲染管线 (Phase 3)

Phase 3 的核心技术突破是 **3DGS 实时渲染**——将 3D 高斯泼溅模型部署到 L40S GPU 上，实现 1080p@30fps 的实时照片级渲染。

```
推理时 (per stream):

  音频输入 (PCM 24kHz)
      │
      ├─→ CosyVoice2 / GPT-SoVITS ──→ 语音输出
      │
      ├─→ Wav2Lip / MuseTalk ──→ 口型 BlendShape 参数 (52 维)
      │
      ├─→ Emotion Engine ──→ 表情 BlendShape 叠加
      │
      ├─→ Gesture Generator ──→ 手臂/身体关节角度
      │
      └─→ 3DGS Renderer (L40S)
          │  输入: 3DGS 模型 + BlendShape + Joint Angles + Camera
          │  输出: 1080p RGBA 帧 @ 30fps
          │  延迟: < 30ms/frame
          ▼
      ┌──────────────┐
      │ Compositor    │  ← 背景 + 叠加层 (商品卡/优惠券/AI标识)
      │ (GPU)         │
      └──────┬───────┘
             │ RGBA 帧
             ▼
      ┌──────────────┐
      │ FFmpeg Encoder│  ← H.264/H.265 编码
      │ (GPU NVENC)   │
      └──────┬───────┘
             │ RTMP 流
             ▼
         直播平台
```

**3DGS 性能估算 (per L40S 48GB)：**

| 分辨率 | FPS | 每卡并发路数 | 每路 VRAM |
|--------|-----|:----------:|----------|
| 1080p | 30 | 2-4 路 | 8-12 GB |
| 720p | 30 | 4-8 路 | 4-6 GB |

> 对于 5 路小型部署，Phase 3 需要 2 张 L40S 专门跑 3DGS 渲染（每张 2-3 路 1080p）。

### 15.4 Few-shot 音色克隆 (Phase 3)

Phase 2 的 CosyVoice2 full-training 需要 15 分钟录音 + 24 小时训练。Phase 3 升级为 **Few-shot 零样本克隆**：

| 方案 | 输入 | 耗时 | 相似度 | 适用场景 |
|------|------|------|:------:|---------|
| GPT-SoVITS (Phase 3) | 3-10s 音频 | 分钟级（无需训练） | > 0.85 | 快速试播 |
| CosyVoice2 zero-shot (Phase 3) | 3-10s 音频 | 分钟级 | > 0.88 | 正式开播 |
| CosyVoice2 full-training (Phase 2) | 15min 音频 | 24h | > 0.92 | 高要求场景 |

**推荐策略：** Few-shot 作为默认方案（快到分钟级），不满意再升级 full-training。

### 15.5 微表情与动作克隆

**微表情克隆流程：**
```
源视频 (3-5min)
  → OpenFace / MediaPipe 面部关键点检测
  → Action Unit (AU) 强度序列提取 (17 AU)
  → 统计建模：AU 分布、co-occurrence 模式、节奏模式
  → 推理时：情感标签 + AU 统计模型 → 采样 → BlendShape 映射
```

**动作克隆流程 (Phase 3)：**
```
源视频 (3-5min, 半身/全身)
  → MediaPipe Pose / MMPose 姿态估计
  → 关键点轨迹提取 + 速度/加速度特征
  → Motion VAE / Style Transfer 学习动作风格
  → 推理时：语音节奏 + 关键词 → 风格化动作生成
```

### 15.6 GPU 资源规划 (全人克隆 Phase 3)

| GPU 卡 | 用途 | 型号 | 并发能力 |
|--------|------|------|---------|
| GPU-1 | 3DGS 渲染 (stream 1-3) | L40S 48GB | 2-3 路 1080p |
| GPU-2 | 3DGS 渲染 (stream 4-5) + Compositor | L40S 48GB | 2-3 路 1080p + 合成 |
| GPU-3 | TTS + LLM 推理 | L40S 48GB | CosyVoice2 + Qwen-3-7B，远超 5 路需求 |
| GPU-4 | 克隆训练（离线，按需启动） | A100 80GB 或 L40S×2 | 3DGS 训练 ~4h，GAN 训练 ~3h |

**5 路 Phase 3 总 GPU 需求：3×L40S + 1×A100 (按需/云端)**

## 16. 关键技术决策记录

| # | 决策 | 选项 | 选定 | 理由 |
|---|------|------|:----:|------|
| 1 | 后端语言 | Python vs Go vs Java | **Python** | AI 生态原生、统一技术栈降低成本、FastAPI 异步性能足够；后续高并发热点可局部 Go 重写 |
| 2 | LLM 策略 (Phase 1) | DeepSeek API vs 通义千问 vs 自部署 | **DeepSeek API** | 性价比最高、中文能力强、延迟可控、零硬件投入 |
| 3 | LLM 策略 (Phase 2) | Qwen-3-7B vs DeepSeek-V3-Lite vs ChatGLM | **Qwen-3-7B** | 开源可商用、中文电商适配好、社区活跃 |
| 4 | TTS 引擎 | CosyVoice2 vs FishSpeech vs GPT-SoVITS | **CosyVoice2** | 零样本克隆最优、中文 MOS > 4.2、阿里通义开源 |
| 5 | 推流模式 | 播单 vs 实时合成 | **播单为主** | 成本低 90%、稳定性高、互动回复可插入播单 |
| 6 | 向量数据库 | Milvus vs Pinecone vs Elasticsearch | **Milvus** | 国产开源、性能领先、无需海外服务 |
| 7 | 对象存储 | MinIO vs 阿里云 OSS vs AWS S3 | **MinIO + OSS** | 私有部署用 MinIO 降本；国内部署可选 OSS 降低网络延迟 |
| 8 | 消息队列 | Kafka vs RocketMQ vs Redis Streams | **Kafka** | 业界标准、生态完善、高吞吐 |
| 9 | 服务网格 | Istio vs Linkerd vs 无 | **Istio** | mTLS、流量管理、可观测性、渐进式采用 |
| 10 | GitOps 工具 | ArgoCD vs Flux | **ArgoCD** | UI 友好、多集群管理、回滚方便 |
| 11 | Python 框架 | FastAPI vs Django vs Flask | **FastAPI** | 异步原生、WebSocket 友好、自动 OpenAPI 文档 |
| 12 | 微服务数量 | 16 个服务 | **16** | 按业务领域拆分，避免过度拆分导致运维复杂度 |
| 13 | 互动输出架构 | 仅语音 vs 语音+文字双通道 | **双通道** | 文字声道省去 TTS/渲染/RTMP，成本趋零，覆盖 90% 简单互动；语音仅用于高价值问答，资源利用率最大化 |
| 14 | Agent 架构策略 | 单 Agent vs Multi-Agent | **渐进式演进** | P1 单流水线 (Simple is better) → P2 播控+问答双 Agent → P3 三 Agent + Orchestrator；对标 Johnsmith 但按需渐进，避免过度设计 |
