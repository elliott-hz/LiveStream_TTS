### 1.2 关键设计决策说明

| 决策 | 说明 | 理由 |
|------|------|------|
| **Monorepo + 独立 Docker 构建** | 一个仓库，每个服务独立 Dockerfile | 共享 proto/libs 方便，但部署时按服务独立镜像、独立伸缩 |
| **Proto 在 `libs/proto/` 集中管理** | 所有 .proto 文件放在一个位置，CI 编译为 Python stubs | 避免 proto 分散难以管理，统一版本号 |
| **共享库作为内部 pip 包** | `libs/common/`、`libs/db/`、`libs/kafka/` 都有 `pyproject.toml` | 各服务 `requirements.txt` 中通过 `-e ../libs/common` 或 pip index 安装 |
| **K8s 清单在服务目录内** | 每个服务 `k8s/` 目录包含自己的 deployment/service/hpa | 服务与部署清单耦合，改动同一 PR 即可 |
| **Helm Chart 在 `deploy/helm/`** | 通用 Chart 模板，各服务通过 values 差异 | 减少重复，统一部署行为（滚动更新策略、健康检查路径等） |
| **Kustomize Overlays 在 `deploy/k8s/overlays/`** | dev/staging/production 三层 | 环境差异通过 overlay 覆盖，不改 base |

## 2. 独立构建与容器化方案

### 2.1 Docker 构建策略

**核心原则：每个服务独立构建为独立 Docker 镜像。** 构建上下文 = 仓库根目录（为了访问 `libs/`），但通过 `.dockerignore` 排除无关文件。

**典型 Dockerfile（以 user-svc 为例）：**

```dockerfile
# services/user-svc/Dockerfile

# === Stage 1: 构建阶段 ===
FROM python:3.12-slim AS builder

WORKDIR /app

# 安装构建依赖
RUN apt-get update && apt-get install -y --no-install-recommends \
    gcc libpq-dev && rm -rf /var/lib/apt/lists/*

# 复制共享库（利用 Docker 缓存）
COPY libs/common/ libs/common/
COPY libs/db/ libs/db/
COPY libs/proto/ libs/proto/

# 复制本服务代码
COPY services/user-svc/ services/user-svc/

# 安装依赖
RUN pip install --no-cache-dir -r services/user-svc/requirements.txt

# === Stage 2: 运行阶段 ===
FROM python:3.12-slim AS runtime

WORKDIR /app

# 从 builder 复制已安装的包
COPY --from=builder /usr/local/lib/python3.12/site-packages /usr/local/lib/python3.12/site-packages
COPY --from=builder /app/services/user-svc/ /app/services/user-svc/
COPY --from=builder /app/libs/ /app/libs/

# 健康检查
HEALTHCHECK --interval=30s --timeout=3s --start-period=10s --retries=3 \
  CMD python3 -c "import urllib.request; urllib.request.urlopen('http://localhost:8000/health')"

# 启动
CMD ["python3", "-m", "services.user-svc.src.main"]
```

**多阶段构建优势：**
- Stage 1 安装编译依赖（gcc、libpq-dev），Stage 2 运行时不含它们 → 镜像缩小 40-60%
- 共享库（libs/）没变时复用缓存，构建仅需几秒
- 每个服务镜像独立，无冗余依赖

### 2.2 构建上下文与 .dockerignore

```dockerfile
# .dockerignore（仓库根目录）
**/__pycache__
**/*.pyc
**/.git
**/.github
**/tests
**/node_modules
.venv
.env
*.md
!.dockerignore
```

**注意：** 构建上下文始终设为仓库根目录，但 `docker build -f services/user-svc/Dockerfile .` 中 `.` 是根目录。`docker build` 命令如下：


### 2.5 批量构建（Docker Bake）

```hcl
# deploy/docker/docker-bake.hcl

  targets = ["gateway-svc", "user-svc", "product-svc", "script-svc",
             "live-mgr-svc", "avatar-svc", "voice-svc", "knowledge-svc",
             "tts-svc", "nlp-svc", "interact-svc", "platform-sync-svc",
             "render-svc", "stream-svc", "analytics-svc", "billing-svc",
             "audit-svc", "profile-svc"]

target "gateway-svc" {
  context = "../.."
  dockerfile = "services/gateway-svc/Dockerfile"
  tags = ["registry.internal/ai-livestream/gateway-svc:${TAG}"]
}

target "user-svc" {
  context = "../.."
  dockerfile = "services/user-svc/Dockerfile"
  tags = ["registry.internal/ai-livestream/user-svc:${TAG}"]
}
# ... 其他服务类似
```

**CI 中使用：**
```bash
cd deploy/docker
TAG=1.0.0-$(git rev-parse --short HEAD) docker buildx bake --push
```

### 2.6 共享库的 pip 安装模式

| 模式 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| `requirements.txt` 中 `-e ../../libs/common` | 简单，依赖在源码中 | Docker 构建上下文需要包含 libs/ | 开发 + Docker 构建 |
| 发布到内部 PyPI（Nexus/DevPI） | 版本管理严格 | 需要维护私有包仓库 | Phase 2/3 |
| CI 预编译为 wheel | 安装快、可缓存 | 构建流水线复杂 | 生产构建 |

**Phase 1 推荐方案：** 使用 `-e ../../libs/common` 相对路径方式。Docker 构建上下文设置为仓库根目录即可。

### 3.2 执行顺序基本原则

1. **基础设施优先**：`libs/proto` → `libs/common` → `libs/db` → CI/CD 流水线 → K8s 集群搭建
2. **无依赖服务优先**：`tts-svc`（核心 AI 能力，仅依赖 libs 基础设施）
3. **被依赖服务先做**：`user-svc`（被所有服务依赖的认证）→ `product-svc` → `script-svc` → `live-mgr-svc`
4. **核心闭环串联**：`gateway-svc`（统一入口）→ `interact-svc`（核心运行时）→ `stream-svc`（输出）
5. **平台适配器收尾**：`platform-sync-svc` 在业务服务就绪后对接
6. **并行开发**：TTS 引擎升级与核心业务服务可并行（分别由不同开发人员负责）

| 文档字符串 | Google Style Docstring |

### 5.4 Proto 设计规则

```protobuf
// 规则 1：使用版本化包名
package tts.v1;

// 规则 2：service 命名 = 业务名 + Service
service TTSService {
  // 规则 3：RPC 命名 = 动词 + 名词（Create/Get/List/Update/Delete）
  rpc SynthesizeStream(SynthesizeRequest) returns (stream AudioChunk);
  rpc GetVoice(GetVoiceRequest) returns (Voice);
  rpc ListVoices(ListVoicesRequest) returns (ListVoicesResponse);
}

// 规则 4：请求/响应消息使用 Request/Response 后缀
message SynthesizeRequest {
  // 规则 5：字段编号 1-15 用于高频字段（使用 varint 编码仅 1 字节）
  string text = 1;        // 高频
  string voice_id = 2;    // 高频
  string emotion = 3;     // 高频
  float speed = 4;        // 高频
  string request_id = 5;  // 高频
  
  // 规则 6：16+ 字段编号留给低频/可选字段
  LiveContext live_context = 16;
  bool skip_cache = 17;
}

// 规则 7：枚举类型从 0 开始
enum Emotion {
  EMOTION_UNSPECIFIED = 0;
  EMOTION_NEUTRAL = 1;
  EMOTION_EXCITED = 2;
  EMOTION_WARM = 3;
  EMOTION_ENTHUSIASTIC = 4;
  EMOTION_CALM = 5;
  EMOTION_URGENT = 6;
}
```

### 6.4 CI/CD Pipeline 配置

```yaml
# .github/workflows/ci-pipeline.yml
name: CI Pipeline

on:
  push:
    branches: [main, 'feature/**', 'fix/**']
  pull_request:
    branches: [main]

jobs:
  # Job 1: 代码质量检查
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: '3.12'
      - run: pip install ruff mypy
      - run: ruff check .        # Lint
      - run: ruff format --check .  # 格式检查

  # Job 2: 仅运行变更服务的测试
  test-changed:
    runs-on: ubuntu-latest
    outputs:
      changed-services: ${{ steps.filter.outputs.changes }}
    steps:
      - uses: actions/checkout@v4
      - uses: dorny/paths-filter@v3
        id: filter
        with:
          filters: |
            tts-svc: ['services/tts-svc/**', 'libs/**']
            user-svc: ['services/user-svc/**', 'libs/**']
            gateway-svc: ['services/gateway-svc/**', 'libs/**']
            # ... 其他服务类似
      - name: Test tts-svc
        if: steps.filter.outputs.tts-svc == 'true'
        run: |
          cd services/tts-svc
          pip install -r requirements.txt
          pytest tests/ -v --cov=src --cov-report=term

  # Job 3: 构建变更服务的 Docker 镜像
  build:
    needs: test-changed
    runs-on: ubuntu-latest
    strategy:
      matrix:
        service: ${{ fromJson(needs.test-changed.outputs.changed-services) }}
    steps:
      - uses: actions/checkout@v4
      - name: Build Docker Image
        run: |
          docker build \
            -t ${{ vars.REGISTRY }}/${{ matrix.service }}:${{ github.sha }} \
            -f services/${{ matrix.service }}/Dockerfile \
            .
      - name: Push to Registry
        run: |
          docker push ${{ vars.REGISTRY }}/${{ matrix.service }}:${{ github.sha }}

  # Job 4: 部署到 Staging（main 分支自动触发）
  deploy-staging:
    needs: build
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Update K8s manifests with new image tag
        run: |
          cd deploy/k8s/overlays/staging
          kustomize edit set image ${{ vars.REGISTRY }}/tts-svc=${{ github.sha }}
      - name: Commit and push
        run: |
          git config user.name "CI Bot"
          git config user.email "ci@ai-livestream.com"
          git add .
          git commit -m "chore: update staging image tags [skip ci]"
          git push
      # ArgoCD 自动检测 Git 变更并同步到 Staging 集群
```
