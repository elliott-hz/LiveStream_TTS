## 1. 自研总体策略：渐进式替代，业务不中断

### 1.1 四条核心原则

```
┌─────────────────────────────────────────────────────────────────┐
│                    ★ 四条铁律 ★                                  │
│                                                                  │
│  1. 业务永远不中断                                                │
│     → 商汤继续跑存量，自研先跑增量，验证后逐步切存量               │
│                                                                  │
│  2. 每个 Phase 都要交付可用的东西                                  │
│     → 不做"大爆炸"式开发，3 个月要看到东西在跑                     │
│                                                                  │
│  3. 成本从低到高，先验证再投入                                     │
│     → POC 阶段 ¥3,000-5,000/月，零硬件投入                       │
│     → 确认跑通后才采购 GPU                                        │
│                                                                  │
│  4. 功能不低于商汤，体验超越商汤                                   │
│     → 形象克隆先复用第三方 API                                    │
│     → 互动体验从一开始就要比商汤好（这是我们最大的差异化）         │
└─────────────────────────────────────────────────────────────────┘
```

### 1.2 四阶段总览

```
Phase 1: POC 验证（1-3 个月）
  │  全云化 API · 零 GPU · 1-3 路试点
  │  月成本 ¥3,000-5,000  |  1 人
  │
  ▼
Phase 2: 功能对齐（3-6 个月）
  │  部分 GPU 自建 · 10 路 · 新客户先行
  │  月成本 ¥1-2 万  |  2-3 人
  │
  ▼
Phase 3: 规模化（6-12 个月）
  │  大部分 GPU 自建 · 100 路 · 存量迁移启动
  │  月成本 ¥5-12 万  |  4-6 人
  │
  ▼
Phase 4: 全量（12-18 个月）
  │  完全自建 · 1000+ 路 · 完全替换商汤
  │  月成本 ¥25-55 万（含团队）  |  8-15 人
```

### 1.3 过渡期双轨运行策略

```
                    Phase 1         Phase 2         Phase 3         Phase 4
                   (1-3月)         (3-6月)         (6-12月)        (12-18月)

商汤存量客户     ████████████████ ████████████████ ████████░░░░░░░ ░░░░░░░░░░░░░░
（~1000 个）      100% 商汤         100% 商汤       逐步迁移        全部迁移完成

自研新客户       ░░░░░░░░░░░░░░ ████████████████ ████████████████ ████████████████
（增量）          1-3 路验证      全部新客走自研   全部新客走自研   全部新客走自研

## 2. Phase 1：POC 验证（1-3 个月，1-3 路）

### 2.1 目标

> **证明自研方案在技术上是可行的，体验上比商汤好。**  
> 跑通完整链路：弹幕 → AI 理解 → 生成回复 → TTS 语音 → 口型渲染 → RTMP 推流  
> 选择 1-3 个新客户（或内部测试直播间）试点，不走存量客户。

### 2.2 技术方案

**策略：全云化，零 GPU，零硬件投入**

```
┌─────────────────────────────────────────────────────────┐
│                   POC 架构（全云化）                       │
│                                                          │
│  阿里云 ECS 24C 48G（一台）                               │
│  ├── interact-svc   ──→ DeepSeek API（LLM 对话）          │
│  │                   ──→ nlp-svc（0.5B 分类，CPU）        │
│  │                   ──→ knowledge-svc ──→ DashVector     │
│  ├── tts-svc        ──→ 阿里云 CosyVoice API（TTS 流式）  │
│  ├── render-svc     2D Viseme（音素→口型，CPU）            │
│  ├── stream-svc     FFmpeg RTMP 推流（CPU）               │
│  └── 管理微服务      user/product/script/live-mgr（CPU）   │
│                                                          │
│  阿里云（全托管中间件）                                     │
│  ├── RDS PostgreSQL  ── 核心数据                          │
│  ├── Redis           ── 缓存 + Session                   │
│  ├── Kafka           ── 异步事件                          │
│  ├── OSS             ── 素材存储                          │
│  └── DashVector      ── 知识库向量检索                    │
│                                                          │
│  外部 API                                                │
│  ├── DeepSeek API    ── LLM 对话生成                      │
│  └── 阿里云 TTS API   ── 语音合成（流式）                  │
└─────────────────────────────────────────────────────────┘
```

### 2.3 交付范围

| 模块 | 功能 | 对标商汤 |
|------|------|:---:|
| **TTS 引擎** | 阿里云 CosyVoice API，5 种情感，流式输出，首包 <300ms | ✅ 持平 |
| **NLP 服务** | 12 类意图识别 + 5 类情感 + 敏感词双模检测 | ⬆️ 超越 |
| **LLM 对话** | DeepSeek API + 带货场景 Prompt 模板 | ⬆️ 超越 |
| **RAG 知识库** | 商品信息向量化 + DashVector 检索 | ⬆️ 超越 |
| **用户画像** | Redis 存储用户标签/购买历史/行为轨迹 | ⬆️ 新增 |
| **文字场控** | AI 自动发评论/弹券/氛围管理/简单问题文字回复 | ⬆️ 新增 |
| **播单引擎** | 脚本片段编排 + 互动插入 + 循环规则 | ✅ 持平 |
| **2D Viseme 渲染** | 音素→BlendShape 口型同步（虚拟形象，CPU） | ⚠️ 略低 |
| **RTMP 推流** | FFmpeg 多路 RTMP 推流 | ✅ 持平 |
| **管理后台** | 商家自助：脚本编排、商品管理、开播控制 | ✅ 持平 |

> **POC 不包含**（Phase 2 补）：真人级形象克隆、Wav2Lip 口型渲染（需 GPU）、自部署 LLM

### 2.4 时间线（12 周）

```
Week 1-2   TTS 引擎接入
           ├── 阿里云 CosyVoice API 流式集成
           ├── 情感标签映射（excited/warm/enthusiastic/calm/urgent）
           └── 音频缓存（高频话术预缓存，降 API 成本 60%+）

Week 3-4   AI 互动引擎
           ├── DeepSeek API 集成 + 带货场景 System Prompt 模板
           ├── NLP 意图/情感/敏感词（0.5B CPU 模型）
           └── RAG 知识库（商品信息向量化 + DashVector）

Week 5-6   用户画像 + 文字场控
           ├── 用户画像（标签/历史/行为 → Redis）
           ├── 文字场控引擎（评论发送/弹券/氛围/负面管控）
           └── 声道路由决策（语音 vs 文字双通道）

Week 7-8   渲染 + 推流
           ├── 2D Viseme 口型渲染（音素→BlendShape，CPU）
           ├── 播单引擎（脚本编排 + 互动插入 + 循环）
           └── FFmpeg RTMP 多路推流

Week 9-10  管理后台 + 全链路打通
           ├── 商家自助后台（脚本/商品/开播管理）
           └── 端到端集成：弹幕→AI→TTS→渲染→推流

Week 11-12 试点验证
           ├── 1-3 个真实直播间试点（新客户或内部测试）
           ├── 对比指标：延迟/稳定性/客户反馈
           └── POC 总结报告 + Phase 2 详细计划调整
```

| 项目 | 月费 | 说明 |
|------|-----|------|
| ECS 24C 48G | ¥350-700 | 包年更优惠 |
| RDS PostgreSQL 2C4G | ¥300-500 | 入门规格 |
| Redis 1GB | ¥150-250 | — |

### 2.6 验收标准

```
□ 弹幕到语音回复端到端延迟 < 3s
□ 弹幕到文字回复延迟 < 1.5s
□ TTS 首包 < 300ms
□ 1-3 路并发稳定运行 72 小时不中断
□ 文字场控自动弹券/弹袋正常工作
□ 用户画像驱动的个性化回复明显优于商汤 FAQ 模式
□ 商家后台可自助完成：创建脚本 → 配置商品 → 开播
```

## 3. Phase 2：功能对齐（3-6 个月，10 路，新客户先行）

### 3.1 目标

> **功能不低于商汤，互动体验显著超越商汤。**  
> 所有新客户走自研平台。支持 10 路并发。补齐形象克隆能力，升级渲染管线。

### 3.2 技术方案

**策略：核心模块 GPU 自建（TTS），渲染升级**

```
┌─────────────────────────────────────────────────────────────┐
│                 Phase 2 架构                                  │
│                                                               │
│  GPU 服务器（新增 2×L40S 48GB）                                │
│  ├── TTS 引擎（CosyVoice2 + TensorRT，本地推理）               │
│  │   - 替换阿里云 TTS API                                    │
│  │   - 首包 < 200ms，单卡 30+ 路                              │
│  │   - 5 情感 + 3 风格                                       │
│  ├── 音频 DSP（降噪 + EQ + 压缩 + 响度归一）                   │
│  └── 3 轨混音（人声 + BGM + 音效）                            │
│                                                               │
│  CPU 服务器（ECS 24C 48G × 1-2）                              │
│  ├── LLM（DeepSeek API 为主，高频简单问题本地 0.5B）           │
│  ├── NLP + RAG（同 POC，优化性能）                            │
│  ├── render-svc（2D Viseme 为主，可选 Wav2Lip POC）           │
│  ├── stream-svc（FFmpeg 10 路推流）                           │
│  └── 管理微服务                                                │
│                                                               │
│  形象克隆（第三方 API 过渡方案）                                │
│  ├── 选项 1：继续短期复用商汤克隆 API（如合同允许）             │
│  ├── 选项 2：接入第三方克隆服务                                │
│  └── Phase 3：自研克隆管线                                    │
└─────────────────────────────────────────────────────────────┘
```

### 3.3 关键补齐能力

| 新增/升级 | 说明 | 对标商汤 |
|----------|------|:---:|
| **CosyVoice2 自建 TTS** | 本地 GPU 推理，替换阿里云 API | ⬆️ 超越 |
| **形象克隆（过渡）** | 第三方 API / 短期复用商汤克隆 API | ✅ 持平 |
| **音色克隆** | CosyVoice2 Speaker Encoder | ✅ 持平 |
| **Wav2Lip 口型（可选）** | GPU 实时口型同步（真人级） | ✅ 持平 |
| **方言 TTS** | 四川话/东北话（对标特看科技） | ⬆️ 新增 |
| **播单+互动深度整合** | 互动插入播单，播完恢复播单 | ⬆️ 超越 |
| **商家自助克隆** | 商家上传视频 → 自动克隆 → 审核 → 上线 | ✅ 持平 |
| **数据看板** | 直播数据实时大屏（GMV/互动/转化漏斗） | ⬆️ 新增 |

### 3.4 时间线（12 周）

```
Week 1-3   GPU 环境搭建 + TTS 引擎

           ├── 音色克隆（CosyVoice2 Speaker Encoder）

## 6. 数据迁移与客户迁移策略

### 6.1 关键前置问题（需与商汤/章总确认）

```
□ 数字人形象数据：源视频谁上传的？训练好的模型能导出吗？
□ 音色数据：音色 Embedding 能导出吗？还是需要重新录音？
□ 商品 & 脚本：存在哪？能导出吗？什么格式？
□ 历史直播数据：弹幕/回复/GMV 能导出吗？
□ 推流配置：各平台推流地址/推流码在谁手里？
```

### 6.2 三阶段迁移策略

**原则：不强迫迁移，用更好的体验吸引迁移。**

```
Stage A：被动等待 + 体验引流（Phase 2-3）
  ┌────────────────────────────────────────────────────┐
  │  • 新客户强制走自研平台                              │
  │  • 存量客户不变（商汤继续跑）                        │
  │  • 给存量客户看自研的效果 demo                       │
  │  • "用我们的新 AI，回复更智能，免费试用两周"          │
  │  • 对商汤不满的客户优先迁移                          │
  │  • 预计 10-20% 存量主动迁移                          │
  └────────────────────────────────────────────────────┘

Stage B：主动迁移 + 双轨并行（Phase 3-4）
  ┌────────────────────────────────────────────────────┐
  │  • 对每批待迁移客户：                                │
  │    1. 重新克隆数字人（用原始视频；如没有则用商汤    │
  │       API 截取一段重新录）                           │
  │    2. 导出商汤的商品/脚本数据 → 导入自研平台          │
  │    3. 配置推流地址                                   │
  │    4. 商汤 + 自研 并行 1-2 周                        │
  │    5. 数据对比无异常 → 切换                           │
  │                                                     │
  │  • 每批 10-20 个客户，逐步推进                       │
  │  • 保留回滚能力：商汤账号至少保留 1 个月缓冲期        │
  └────────────────────────────────────────────────────┘

Stage C：收尾（Phase 4 后期）
  ┌────────────────────────────────────────────────────┐
  │  • 所有客户完成迁移                                 │
  │  • 商汤合同到期/提前终止                             │
  │  • 历史数据归档                                     │
  └────────────────────────────────────────────────────┘
```

### 6.3 迁移风险预案

| 风险 | 概率 | 应对 |
|------|:---:|------|
| 商汤拒绝数据导出 | 🔴 高 | 保留商家原始素材（视频/商品表），重新克隆+导入。历史弹幕可放弃 |
| 客户拒绝迁移 | 🟡 中 | 不强推，商汤继续跑直到合同到期。用体验差距吸引 |
| 迁移期间出故障 | 🟡 中 | 双轨并行 2 周，问题秒切回商汤 |
| 形象克隆效果不如商汤 | 🟡 中 | Phase 2 评估第三方 API 兜底，Phase 3 自研追赶 |

## 10. 合作模式建议

### 10.1 几种可能模式

| 模式 | 我的角色 | 薪资 | 股权 | 适合场景 |
|------|---------|:---:|:---:|---------|
| **A: 全职 CTO** | 加入公司，带技术团队 | 市场价 | 3-8% | 深度绑定，有融资/上市计划 |
| **B: 技术合伙人** | 联合创始人，持股运营 | 生活费 + 后期补薪 | 10-25% | 公司早期，技术入股为主 |
| **C: 顾问 + 交付** | 外部身份，按项目交付 | 顾问费 / 项目款 | 0-2% | 过渡期，建立信任后再决定 |
| **D: 技术公司独立** | 我成立技术公司，章总出业务 | 按服务收费 | 各自持公司 | 松散耦合，风险隔离 |

### 10.2 建议路径

```
建议：C → A/B

第一阶段（3-6 个月）：模式 C
  - 我以外部技术顾问身份完成 POC + Phase 2
  - 交付费：协商（如 ¥3-5 万/月 或 固定项目费）
  - 双方评估：技术落地能力 vs 业务能力
  - 6 个月后双方都有充分信息决定是否深度绑定

第二阶段（6 个月后）：模式 A 或 B
  - 基于 6 个月的信任 + 实际结果
  - 谈股权 + 薪资更客观，不是画饼
  - 此时已有跑起来的系统，技术估值有依据
```
