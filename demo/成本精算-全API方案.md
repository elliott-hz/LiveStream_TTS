## 二、NLP 本地 CPU 方案

### 2.1 两个任务分阶段处理

```
弹幕到达 → Stage 1: 敏感词过滤 → Stage 2: 意图分类 → 路由决策
              AC 自动机             FastText / albert-tiny
              ~0.01ms/条            ~0.5-5ms/条
              几乎免费              轻量 CPU
```

| 模型 | 大小 | 单条推理 | 单核 QPS | 12 核 QPS |
|------|:---:|:---:|:---:|:---:|
| FastText | ~10MB | ~0.5ms | 2,000 | **24,000** |
| albert-chinese-tiny | ~16MB | ~5ms | 200 | **2,400** |

### 2.2 5 秒批处理窗口

不逐条处理弹幕，而是每 5 秒收一批 → 批量推理 → 路由分发。好处：
- **同类弹幕合并去重**：3 个人问"多少钱" → 只调 1 次 LLM，回复推给 3 人
- **LLM 调用更少**：批内去重能再省 15-20% LLM 费用
- **上下文更好**：一批弹幕一起给 LLM，LLM 知道"这段时间大家都在问尺码"

```
5秒收集窗口（500路×5条/min → 每5秒约208条）
  ├── "有XL吗"     ┐
  ├── "多少钱"      │ NLP 批量分类
  ├── "主播好漂亮"   │ → 闲聊×1, 提问×3, 下单×1
  ├── "XL码链接"    │ → 同类合并: "尺码相关" ×2
  ├── "已下单"      │ → 只调 3 次 LLM (不是 5 次)
  └── "什么面料"    ┘
```

### 2.3 算力验证：3×4C8G 够不够？

**大促峰值（最坏情况）**：500路 × 100条弹幕/min × 每 5 秒 = **4,167 条/批**

| 阶段 | 模型 | 单核耗时 | 12 核并行耗时 | 结论 |
|------|------|:---:|:---:|:---:|
| 敏感词 | AC 自动机 | ~40ms | ~3ms | ✅ |
| 意图分类 | FastText | 2.1s | **~175ms** | ✅ |
| 意图分类 | albert-tiny | 5.2s | **~435ms** | ✅ |

> 5 秒批处理窗口内 175ms 就算完了，距离窗口关闭还有 4.8 秒。**3×4C8G 不仅够 500 路，实际能撑到 2000 路大促峰值（16,667 条/批，FastText 12 核 ~700ms）。** 三台冗余部署，挂一台不影响服务。

## 三、口型同步（Wav2Lip）技术方案

### 3.1 全云化部署架构

```
TTS 语音 → Wav2Lip 推理服务 (云 T4 GPU) → 口型视频帧 → FFmpeg 合成推流
```

- Wav2Lip Docker 容器部署在云 T4 竞价实例上
- TTS 生成语音后，内网 gRPC 调用 Wav2Lip 服务
- 返回口型匹配的视频帧序列，stream-svc 混入推流

### 3.2 GPU 规模计算

```
每路：
  语音回复: 5条弹幕/min × 30% = 1.5 次/min
  每次 Wav2Lip 推理: ~3s T4 GPU 时间（3-4s 音频 → 口型帧）
  GPU 占用: 1.5 × 3s = 4.5s/min → 每路需 ~0.075 T4 并发

每张 T4：可服务 ~13 路（60s / 4.5s，80% 利用率）
```

| | 200路 | 500路 | 1000路 | 2000路 |
|------|:---:|:---:|:---:|:---:|
| T4 并发需求（理论） | 15 | 38 | 75 | 150 |
| T4 实际部署（80%利用率） | **12** | **30** | **60** | **120** |

## 十、边际成本递减

每新增一个客户，增量成本只有云+API+GPU（研发不增加）：

| 新增区间 | 边际成本/客户/年 | 说明 |
|------|:---:|------|
| 200 → 500 | ~¥10,700 | 视频解析 API 占比大，摊薄效果弱 |
| 500 → 1000 | ~¥11,400 | 规模效应微弱 |
| 1000 → 2000 | ~¥10,800 | 几乎恒定 |

边际成本稳定在 **¥11,000/年左右**。视频解析量提升后，边际成本接近甚至略高于供应商 ¥10,000/年。全 API 方案的线性成本问题更加突出。

### 为什么每客户成本几乎不降？

拆开看 500 路 → 2000 路：

| | 500路/每客户 | 2000路/每客户 | 变化 |
|------|:---:|:---:|:---:|
| 云服务器 | ¥168 | ¥142 | ↓ 15%（摊薄了） |
| API+GPU | ¥783 | ¥783 | **∠ 一动不动** |
| **合计** | **¥951** | **¥925** | ↓ 3%（几乎没降） |

视频解析 + 弹幕都是按量线性计费。视频解析 API 占比从之前的 14% 提升到 31%，线性成本更加顽固。**规模化降的是服务器成本（占 18%），降不到 API 成本（占 82%）。**

做大规模之后，必须把 TTS、ASR、场景检测从 API 切换到自有 GPU，否则线性成本永远跑不赢。

### 11.6 结论

| 方案 | 每客户/年（1000客户） | 相比供应商 | 关键约束 |
|------|:---:|:---:|------|
| 供应商 | ¥10,000 | — | 零自主权，线性成本 |
| 全 API | ¥11,390 | **+¥1,390** ⚠️ | 比供应商还贵 |
| 全 API 优化后 | ¥8,920 | −¥1,080 | 仅靠缓存/遮挡勉强打平 |
| **GPU 自建** | **¥4,030** | **−¥5,970** | 需 GPU 运维能力，前期需验证 |

**视频解析量修正后，结论更加明确：全 API 方案不成立——每客户成本 ¥11,390，比供应商还贵 14%。GPU 自建方案是唯一可行的路线，每客户 ¥4,030，比供应商便宜 60%。** 核心原因：视频解析 API（ASR ¥3.5 + 场景检测 ¥7.5 = ¥11/视频）叠加每天 1 个视频的使用频率后，占到了 API 总费用的 31%，这部分只有切 GPU 才能消化。
