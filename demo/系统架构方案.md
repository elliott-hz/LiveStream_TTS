# 系统架构方案

> **基于**：Demo v3 — 预录视频 + AI 互动 + 多平台知识库
> **规模目标**：5000 视频/周处理 · 2000 路同时推流
> **原则**：Phase 2 轻量级，零 GPU 依赖；Phase 3 可扩展数字人渲染

---

## 一、整体架构

> 核心业务服务层（前端→网关→业务中台→微服务）对两个方案**完全相同**。

## 二、服务职责

| 服务 | 职责 | 关键依赖 |
|------|------|------|
| **gateway-svc** | HTTP→gRPC 代理、JWT 鉴权、限流、WebSocket 弹幕推送 | — |
| **platform-svc** | 全部业务逻辑（11 个模块），REST API 的 gRPC 实现 | PostgreSQL, Redis |
| **knowledge-svc** | 知识库 RAG 检索：商品详情向量化 → 弹幕语义匹配 → 返回相关片段 | Milvus, Embedding API |
| **interact-svc** | 互动管线：弹幕→意图分类(nlp-svc)→知识检索(knowledge-svc)→**调用 LLM API 生成文字回复**→(可选)**调用 TTS API 合成语音** | nlp-svc, knowledge-svc, LLM API, TTS API |
| **nlp-svc** | 意图分类（提问/闲聊/下单/投诉）、敏感词过滤 | Option A: 阿里云 NLP API / 百度 AI API · Option B: 本地轻量模型 (CPU) |
| **tts-svc** | TTS 语音合成适配层：管理音色、参数（语速/音量/语调）、调用底层 TTS 引擎 | Option A: 阿里云 CosyVoice API / Option B: CosyVoice2 本地 GPU |
| **stream-svc** | FFmpeg 播单循环推流 + 直播时 TTS 音频混入推流音频通道 | FFmpeg |
| **video-parser-svc** | **视频解析（异步服务）**：商户 SPA 上传视频 → platform-svc 触发 → video-parser-svc 异步处理（ASR→场景检测→切割→脚本→SKU）→ 回调通知商户端 | Option A: 阿里云 API / Option B: Whisper+CLIP GPU / Option C: 商汤如影 |

## 四、核心数据流

### 4.1 视频素材处理（离线，一次性）

> 由 **video-parser-svc** 负责，以 Kubernetes Job 方式异步执行。

```
商户上传视频(2h) → OSS 存储 → 触发解析 Job
  │
  ├──→ Step 1: ASR 语音识别 → 全文稿 + 逐句时间戳
  │         Option A: 阿里云 ASR API     (~3min,  ¥3.5)
  │         Option B: Whisper large-v3    (~6min,  GPU本地)
  │
  ├──→ Step 2: 场景/商品边界检测 → 切分时间点列表
  │         Option A: 阿里云智能媒体 API   (~15min, ¥7.5, 准确率~70%)
  │         Option B: CLIP ViT-L/14       (~10min, GPU本地, 准确率~90%)
  │
  ├──→ Step 3: LLM 脚本清洗 → 按商品分段 + 去口语词 + 补全
  │         DeepSeek API                  (~30s,   ¥0.001)
  │
  ├──→ Step 4: FFmpeg 无损切割 → 独立切片视频文件
  │         ffmpeg -c copy                (~2min,  CPU, 纯磁盘IO)
  │
  └──→ Step 5: TTS 音色提取（可选, 按商户）
           Option A: CosyVoice API Few-shot (~5min, ¥1)
           Option B: CosyVoice2 GPU Few-shot (~5min, GPU本地)

输出 → 切片视频 + 脚本文稿 + SKU 关联 + TTS 音色 → 写入 DB + OSS
  → 商户手动微调（±1s/±5s、修正SKU、编辑脚本）
  → 发布到素材库
```

### 4.2 直播推流（实时）

```
商户创建直播间 → 挂载直播视频 → 设置排班
  → 开播
  → stream-svc 按播单循环播放切片视频
  → FFmpeg RTMP 推流到淘宝/抖音/快手
  → 直播中控台实时监控（画面、弹幕、数据）
```

### 4.3 AI 互动回复（实时）

> 由 **interact-svc** 编排，**LLM API** 和 **TTS API/tts-svc** 是核心引擎。

```
观众弹幕(WebSocket/平台API)
  │
  ▼
interact-svc（编排层）
  │
  ├──→ nlp-svc        意图分类（提问/闲聊/下单/投诉）
  ├──→ knowledge-svc   RAG 检索相关知识片段（Milvus 向量搜索）
  │
  ├──→ LLM API（核心推理引擎）
  │     System Prompt + 知识片段 + 弹幕 → 生成文字回复
  │     DeepSeek / 通义千问 / GPT
  │     延迟: ~1-2s
  │
  └──→ TTS（可选，核心语音引擎）
        Option A: 阿里云 CosyVoice API → 文字→语音
        Option B: tts-svc → CosyVoice2 GPU 本地推理
        延迟: ~0.5-1s

  → 文字回复通过平台 API 发送到聊天区
  → 语音回复 → stream-svc 混入推流音频通道（替换原视频音轨片刻）
```

> **LLM 和 TTS 都不在业务服务内**。interact-svc 只做编排，实际推理由外部 API 或独立的 tts-svc 完成。

## 六、两个独立模块：各自 Option A / B

视频解析和 TTS 是**完全独立**的两个模块，各自选择技术路线，互不影响。

### 6.1 模块一：视频解析

> 离线 Job，输入原始视频 → 输出切片+脚本。**一次性的**，解析完就不再需要 GPU/API。

| | Option A：全 API | Option B：GPU 本地推理 | Option C：商汤（如开放） |
|------|------|------|------|
| **ASR** | 阿里云 ASR API | Whisper T4 | 商汤内部 |
| **场景检测** | 阿里云智能媒体 API | CLIP T4 | 商汤内部 |
| **脚本清洗** | DeepSeek API | DeepSeek API | 未知 |
| **视频切割** | FFmpeg CPU | FFmpeg CPU | 商汤内部渲染 |
| **TTS 音色提取** | CosyVoice API | CosyVoice2 GPU | 商汤内部 |
| **准确率** | ~70% ⚠️ | ~90% ✅ | ~95% ✅（推测） |
| **单视频耗时** | ~25min | ~21min（GPU） | ~10h ⚠️（含数字人训练） |
| **单视频费用** | ¥11 | ¥0.001 | 未知 |
| **5000视频/周GPU** | 零 | 10-14×T4 | 未知（商汤云端） |
| **5000视频/周费用** | ~¥55,000 | 硬件折旧 | **未知**（商务定价） |
| **架构依赖** | 调外部 API | video-parser-svc + GPU | 商汤云端 → 推送到客户端 |
| **额外产出** | 切片+脚本+TTS音色 | 切片+脚本+TTS音色 | + 数字人模型（可实时渲染） |

> Option C 的 10h 包含了数字人模型训练（3D重建+口型对齐+神经渲染），不只是视频切分。如果商汤开放"仅切分"API（不做数字人训练），耗时应该能降到 1-2h。

### 6.2 模块二：TTS 语音合成

> 直播时实时调用，输入文字 → 输出语音。**按次调用**，与直播路数和弹幕量正相关。

| | Option A：TTS API | Option B：GPU 本地推理 | Option C：商汤（如开放） |
|------|------|------|------|
| **通用 TTS** | 阿里云 CosyVoice API | CosyVoice2 T4 | 商汤内部 |
| **音色克隆** | 阿里云 API，¥1/次 | CosyVoice2 T4，5min | 商汤方案（含视频形象） |
| **精调音色** | 阿里云定制，¥500/次 | GPT-SoVITS T4，90min | 商汤方案 |
| **延迟** | ~500ms | ~300ms | 未知 |
| **并发** | API 不限 | 受 GPU 限制 | 未知 |
| **2000路直播** | 零 GPU | 2-4×T4 | 未知 |
| **费用** | ~¥24,000/月 | 硬件折旧 | **未知** |
| **附加价值** | 纯语音 | 纯语音 | **语音+数字人口型同步** |

> Option C 的独特价值：商汤 TTS 和数字人模型是联动的——TTS 语音直接驱动数字人口型实时渲染。Option A/B 只有语音没有口型。

### 6.3 组合建议

| 阶段 | 视频解析 | TTS | 理由 |
|------|:---:|:---:|------|
| **PoC** | A（API） | A（API） | 零硬件，快速验证 |
| **过渡期** | B（GPU） | A（API） | 视频解析量大先迁 GPU，TTS 量小继续 API |
| **规模化** | B（GPU） | B（GPU） | 两端都 GPU，三年 6 倍成本优势 |
| **Phase 3 数字人** | C（商汤） | C（商汤） | 需要口型同步+实时渲染时，商汤整体方案 |

> C 的决策取决于：商汤是否开放 API、定价是否合理（vs 自建 Wav2Lip）、是否需要数字人口型同步。如果只是视频切分+TTS，B 方案已足够。

## 七、规模测算

### 7.1 视频解析：5000 视频/周

| 步骤 | Option A | GPU | Option B | GPU |
|------|:---:|:---:|------|:---:|
| ASR | 阿里云 API | 0 | Whisper T4 | 500h |
| 场景检测 | 阿里云 API | 0 | CLIP T4 | 833h |
| 脚本清洗 | DeepSeek API | 0 | DeepSeek API | 0 |
| 视频切割 | FFmpeg CPU | 0 | FFmpeg CPU | 0 |
| **GPU 合计** | **0** | | | **1,333h/周 → 10×T4** |
| **周费用** | **~¥55,000** | | | **硬件折旧** |

> Option B 基础解析 10 张 T4。如果要加 TTS 音色提取（CosyVoice2）同机跑，再加 4 张 = 14 张。

### 7.2 TTS 语音：2000 路直播实时

| 场景 | 调用频率 | Option A 月费 | Option B GPU |
|------|:---:|------|:---:|
| 2000路 × 10弹幕/分 × 40%语音回复 | ~8,000 次/分 | 通用 TTS ¥2/千字 ≈ ¥800/天 ≈ **¥24,000/月** | 2-4×T4 |
| 音色克隆（每商户一次） | 一次性 | ¥1/次（Few-shot）| CosyVoice2 本地 |

> TTS 的 GPU 压力远小于视频解析。2000 路直播的语音回复只需 2-4 张 T4 处理峰值。

### 7.3 AI 文字回复：LLM API（两方案相同）

| 场景 | 调用量 | 费用 |
|------|:---:|------|
| 2000路 × 10弹幕/分 | 20,000 次/分 ≈ 333 QPS | DeepSeek 约 ¥0.001/次 |
| 日调用量 | ~2,880 万次 | **~¥28,800/天 → ¥864,000/月** ⚠️ |

> ⚠️ 这是理论峰值。实际 2000 路直播弹幕不会全部触发 LLM 回复：
> - 意图分类过滤掉 30% 闲聊（不回复）
> - 40% 用话术模板精确匹配（不调 LLM）
> - 实际 LLM 调用 ≈ 30% × 2,880 万 = 864 万次/天 → **~¥8,640/天 → ~¥260,000/月**

### 7.4 推流：2000 路同时在线

| 资源 | 单路消耗 | 2000 路总需求 |
|------|:---:|------|
| CPU | ~0.1 核（FFmpeg stream copy） | **200 核** |
| 内存 | ~256MB（FFmpeg buffer） | **512 GB** |
| 上行带宽 | ~4 Mbps（1080p） | **8 Gbps** |
| Pod 数 | 1 Pod/路 | **2000 Pod** |

> stream-svc 按 50 路/Pod 部署 = 40 个 Pod，每 Pod 管理 50 个 FFmpeg 进程。用 Node.js/Go 的轻量协程模型，单 Pod 4 核 8G 可支撑。

### 7.5 AI 互动：并发弹幕

| 假设 | 数值 |
|------|:---:|
| 平均弹幕频率 | 10 条/分钟/直播间 |
| 峰值弹幕频率 | 50 条/分钟/直播间 |
| 2000 路峰值 QPS | 2000×50÷60 = **1,667 弹幕/秒** |
| LLM API 并发 | 需要支持 ~500 并发请求 |
| TTS 并发 | ~200 并发（40% 弹幕触发语音回复） |

> LLM 和 TTS 本身是外部 API 调用，不限并发。瓶颈在 API 网关的 WebSocket 连接数和 interact-svc 的消息处理吞吐。

| 节点类型 | 数量 | 配置 | 用途 |
|------|:---:|------|------|
