# 主数据同步调研 — 商品 SKU 跨平台推送

> **调研日期：** 2026-07-22
> **调研目的：** 梳理各直播电商平台是否支持通过 API 对接商品数据，评估「一个平台集中管理商品，同步推送到各直播电商平台」的技术可行性。
> **结论先行：** 所有主流平台均提供商品管理 API，技术上完全可行。但「同步」不等于「一键复制」——每个平台有独立的类目体系、属性规范和审核流程，实际需要逐平台适配转换。

---

## 一、各平台商品 API 概览

| 平台 | 商品发布 | 商品更新 | 库存/价格同步 | 上下架 | API 门槛 |
|------|:---:|:---:|:---:|:---:|------|
| **淘宝** | ✅ `taobao.item.add` | ✅ `taobao.item.update` | ✅ `taobao.item.quantity.update` | ✅ | 需 ISV 资质 + 店铺 OAuth 授权 |
| **抖音** | ✅ 抖店商品 API | ✅ | ✅ | ✅ | 需企业认证 + 抖店开通 |
| **快手** | ✅ 快手小店商品 API | ✅ | ✅ | ✅ | 需企业认证 + 小店开通 |
| **京东** | ✅ JOS 商品 API | ✅ | ✅ | ✅ | 需企业资质 + 店铺授权 |
| **拼多多** | ✅ `pdd.goods.add` | ✅ `pdd.goods.update` | ✅ `pdd.goods.sku.update` | ✅ | 需企业资质 + 店铺授权 |
| **视频号** | ❌ 仅手机端操作 | ❌ | ❌ | ❌ | 无开放 API |

> 淘宝、抖音、快手、京东、拼多多五家覆盖了 ~78% 的直播电商份额，均具备完整的商品 API 能力。

## 二、核心挑战：不是"推送"，是"转换"

### 2.1 类目体系不互通

同一件「法式碎花连衣裙」在不同平台的类目路径完全不同：

| 平台 | 类目路径 |
|------|------|
| **淘宝** | 女装/女士精品 > 连衣裙 > 碎花裙 |
| **抖音** | 服饰 > 女装 > 裙子 > 连衣裙 |
| **快手** | 女装 > 裙装 > 连衣裙 |
| **京东** | 服饰鞋包 > 女装 > 裙装 > 连衣裙 |
| **拼多多** | 服饰箱包 > 女装 > 连衣裙 |

→ 需要维护「平台类目映射表」，发布时自动匹配。

### 2.2 属性规范不一致

| 属性 | 淘宝 | 抖音 | 京东 |
|------|------|------|------|
| 尺码 | S/M/L/XL (自定义) | S/M/L/XL (平台枚举) | 标准尺码号 (155/80A 等) |
| 材质 | 自由文本 | 需匹配平台材质库 | 需第三方质检报告 |
| 颜色 | 自定义 + 色卡 | 平台标准色 | 平台标准色 + 自定义 |

→ 需要「属性转换引擎」，将我们标准属性映射到各平台格式。

### 2.3 图片规范不同

| 要求 | 淘宝 | 抖音 | 拼多多 |
|------|------|------|------|
| 主图数量 | 5 张 | ≥3 张 | ≥3 张 |
| 白底图 | 第 5 张必须 | 非必须（推荐） | 主图建议白底 |
| 比例 | 1:1 | 1:1 或 3:4 | 1:1 |
| 大小 | ≤3MB | ≤5MB | ≤2MB |

→ 上传时可能需要自动裁切/压缩。

### 2.4 审核规则不同

| 平台 | 敏感特征 |
|------|------|
| **淘宝** | 禁止"最""第一""国家级"等极限词 |
| **抖音** | AI 自动审核 + 人工抽审，标题党易被拒 |
| **拼多多** | 严查低价引流、虚假宣传 |
| **京东** | 3C 需 3C 认证，食品需生产许可证 |

→ 需要「商品描述合规检查」模块，按平台规则过滤违规内容。

## 三、行业现有方案

### 3.1 全渠道 ERP / 商品中台

市面上已有成熟产品解决这个问题：

| 产品 | 接入平台数 | 月费 | 特点 |
|------|:---:|------|------|
| **店小秘** | 20+ | 免费/付费 | 跨境电商为主，国内电商也支持 |
| **马帮 ERP** | 15+ | ¥299起 | 侧重库存和订单管理 |
| **聚水潭** | 30+ | ¥500起 | 国内最主流，对接最深 |
| **旺店通** | 20+ | ¥800起 | 偏高端，品牌客户为主 |

这些产品本质就是「商品数据中台 + 平台适配器」架构。

### 3.2 与我们产品的定位差异

| 维度 | 全渠道 ERP（店小秘等） | 我们的产品 |
|------|------|------|
| **核心能力** | 商品管理 + 订单/库存 | 直播推流 + AI 互动 |
| **商品来源** | 从 ERP 录入或 Excel 导入 | 从平台抓取（AI 解析） |
| **商品去向** | 推送到各平台店铺 | 作为 AI 知识库（互动回复用） |
| **同步方向** | 我们 → 平台 | 平台 → 我们（抓取） |

结合当前产品定位，**商品同步的短期价值是从平台抓取商品详情作为知识库**（已实现），而不是反向推送商品到平台。

## 四、建议方案

### Phase 1（当前）：从平台抓 → 商品知识库 ✅ 已实现

```
淘宝商品链接 → AI 解析 → 商品知识库 → 直播互动回复
```

### Phase 2（中期）：商品主数据管理 + 轻量发布

```
商品知识库（标准商品）→ 属性转换 → 调用平台 API → 发布到单个平台
```

- 先做**单向**：选择商品 + 选择目标平台 → 一键发布
- 不做实时库存同步（太重）
- 定位：帮助商户快速在新平台开店上品

### Phase 3（远期）：全渠道商品中台

```
商品知识库 ↔ 多平台实时同步（库存/价格/上下架）
```

- 需要订单系统配合（库存扣减）
- 需要售后系统配合（退货入库）
- 这一层建议直接集成聚水潭/旺店通而非自研

## 五、对 Demo 的影响

当前 Demo 中，「商品知识库 → 平台接入」已有能力 Tag 展示（🟢淘宝 🟢抖音等）。建议新增一个能力维度：

| 能力 | 说明 |
|------|------|
| **知识库读取** ✅ 已有 | 从平台抓取商品详情作为 AI 互动知识来源 |
| **商品发布** 📋 Phase 2 | 将商品推送到平台店铺（需商品 API 授权） |

技术可行性明确，产品节奏需要你决策。
